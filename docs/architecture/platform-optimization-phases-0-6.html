Background jobs — manual control

Removed automatic Vercel & Supabase cron schedules

Operational control
Before: Production could wake the app on a timer for refunds, classroom archiving, telemetry pruning, AI bot posts, and more — even when no students were active. Supabase pg_cron could also run database jobs automatically.
Before
  • vercel.json listed cron schedules for production
  • Jobs fired without team oversight
  • Gyan bot could post on a timer (AI + database load)
  • Unclear when maintenance actually ran
After
  • vercel.json is empty — no auto schedules in dev or production
  • pg_cron not enabled on production Supabase
  • All maintenance API routes still exist
  • Team triggers jobs manually or via external scheduler when ready
  • CRON_SECRET supported when schedules are re-added

Maintenance routes still available (not on a timer)

RoutePurpose
/api/cron/refund-doubt-bountiesRefund expired doubt bounties (7-day rule)
/api/cron/archive-classroom-sectionsDeactivate classroom sections past schedule_end_date
/api/cron/prune-telemetry-logsDelete old AI logs (90d) and dwell events (180d); ensures dwell partitions exist
/api/cron/flush-site-presenceFlush Redis presence buffer → Postgres (Phase 5; needs Upstash)
/api/cron/warm-admin-analyticsPre-warm admin dashboard cache (Phase 5)
/api/cron/gyan-bot-postAI community bot post — do not schedule without explicit team review

How this helps

Predictable operations. Maintenance runs when it makes sense — not at random hours. Prevents surprise server wake-ups during low-traffic periods. Safe routes (refunds, archiving) can be re-scheduled later; heavy AI jobs stay under human control.

Note for investors: Doubt refunds and classroom archiving no longer happen automatically until the team schedules them again. This is intentional — full visibility over production behavior.

Quick wins — less work per student action

Five targeted code changes; same student experience

Efficiency
Before: Everyday actions — opening profile, staying online, asking Gyan+ — triggered more database writes, realtime reloads, and AI textbook searches than needed for the same UX.
#AreaBeforeAfterWhere changed
1 Study-day penalty reconcile Heavy reconcile could run on every profile/dashboard load Runs at most once per IST calendar day, only when client sends reconcile=1 on first load that day studyDaysClient.ts, study-days/route.ts
2 Site presence heartbeat Browser ping every 20 seconds Ping every 50 seconds (client + server dedupe aligned) sitePresenceConstants.ts, SitePresenceProvider.tsx, site-presence/route.ts
3 Profile Realtime reloads One realtime event → 4 separate data reloads Events batched 450ms → one reload burst StudentProfileHubPanels.tsx
4 Subject chat RAG passages Up to 10 textbook passages per student question Capped at 5 for subject chat (admin/agent routes still use 10–25) gyanContentPolicy.ts, subject-chat/route.ts
5 Modal RAG retriever Up to 3 vector search passes per question (strict → cross-subject → all grades) Skips Pass 2 & 3 when Pass 1 already returns chunks above subject similarity threshold; only falls back if Pass 1 score below 0.28 floor modal-rag/retriever.py (deployed to Modal)

RAG multi-pass logic (simplified)

Old search flow
  • Pass 1: exact class + subject
  • Pass 2: same class, all subjects — often ran even when Pass 1 was fine
  • Pass 3: all classes, all subjects
  • Up to 3 Supabase vector RPC calls per Gyan+ question
New search flow
  • Pass 1: exact class + subject
  • If good chunks found → stop here
  • Pass 2/3 only when Pass 1 is truly weak (< 0.28 relevance)
  • Usually 1 search instead of 3 — faster answers
  • Per-subject thresholds: Maths 0.42, Physics/Chemistry 0.36, Biology 0.30

How this helps

  • Students still get full Gyan+ and subject-chat answers grounded in NCERT textbooks.
  • Buddy “online” status still updates — slightly slower (30s) but still feels live.
  • Profile panels behave the same with fewer background network spikes.
  • Platform load grows slower as daily active users increase.

Infrastructure architecture — two systems, one product

Locked decision: keep app database and AI textbook library separate

Reliability
Decision: EduBlast uses two Supabase projects. We evaluated merging them into one. We chose Option A — keep separate permanently to protect live Gyan+ Q&A.
OptionDecisionRationale
A — Keep separate✅ SelectedTextbook vectors isolated; Modal retriever unchanged; zero migration risk to student AI answers
B — Merge into main❌ RejectedWould require migrating 4,217 chunks and risk breaking production Gyan+ path
C — External vector store⏸ DeferredNot needed at current scale

Two systems — roles

SystemRoleContains
Main SupabaseApp platformAuth (38 users), profiles, classrooms, doubts, posts, lessons, telemetry, storage (~93 MB)
RAG SupabaseAI textbook library onlytextbook_chunks + match_chunks vector search (~70 MB, 4,217 chunks)
Modal sidecarAI embedding & retrievalBGE-M3 model; connects to RAG Supabase only for search
Next.js appStudent & teacher UITalks to main Supabase + Modal; never queries textbook chunks from main DB

Configuration rules (locked)

LayerRule
Modal secret custom-secretRAG_SUPABASE_URL + RAG_SUPABASE_ANON_KEY → RAG project only
Next.js RAG_SIDECAR_URLPoints to Modal only — not main Supabase
Next.js NEXT_PUBLIC_SUPABASE_URLMain project only — never RAG project
RAG secrets on VercelNot stored — live in Modal secrets only
SUPABASE_FETCH_RETRIESRecommended value 1 (fewer retry storms on flaky networks)

Other infrastructure policies

Considered
  • Merge AI textbook DB into main app DB
  • Custom auth domain auth.edublast.in (Google shows Supabase URL today)
  • Supabase preview database branch per pull request
  • Pause or delete RAG project
Policy
  • Never merge, pause, or delete RAG project
  • Custom auth domain deferred — enable when team approves DNS + Pro setup
  • Avoid preview DB branches unless truly needed; delete within 3–7 days if used
  • Git branches = app code only; no linked Supabase preview DB today

How this helps

Zero risk to the AI tutor students already use. Textbook search stays independent and stable. Clear engineering rules prevent accidental misconfiguration. Team can scale Gyan+ via smarter retrieval (Phase 1) without touching the textbook library architecture.

Topic hub UX — Explore shows real content

Product quality fix — not infrastructure; content was in DB but UI looked empty

Product quality
Root causes: (1) Explore only loaded basics level — intermediate/advanced prose ignored. (2) AI agent stores "-" for skipped sections — UI showed blank dashes. (3) hub_scope topic vs chapter are different database rows. (4) Subtopic AI needs all 3 levels viable — admins had no visibility.

What shipped (6 deliverables)

#DeliverableWhat it does
1topicHubDisplay.tsTreats "-" as empty; helpers to merge sections across difficulty levels
2fetchTopicHubDisplayBundleParallel fetch basics + intermediate + advanced; merges best overview text + subtopic previews
3TopicHubOverviewSections.tsxShared Explore UI — honest empty states when previews exist but overview pending
4explore-1/page.tsxUses display bundle instead of basics-only fetch; normalizes placeholders after AI generate/regenerate
5subtopicCompleteness.tsGate viability ignores "-" placeholders — only real prose or previews count
6Admin Topic hub page/admin/topic-hub + API /api/admin/topic-hub-coverage — per-topic status for all 3 levels
Before (student sees)
  • Chapter overview shows — even when DB has content
  • Only “basics” row consulted
  • Subtopic previews ignored when overview sections empty
  • Placeholder "-" treated as real text
  • No admin view of which topics need AI generation
After (student sees)
  • Best available text from any difficulty level shown
  • Clear message: “Subtopic previews ready — overview still being filled”
  • Subtopic preview cards surface when overview prose missing
  • Admin hint: which levels still needed for subtopic AI gate
  • Admin dashboard filters by board, subject, class, scope, topic

Admin coverage badges per level

BadgeMeaning
okLevel has real overview prose or subtopic previews — gate viable
previewsRow exists with subtopic previews but weak overview
emptyRow exists but no viable content
missingNo database row for that level

How this helps

PCM students see meaningful chapter introductions in Explore instead of blank sections. Content team sees exactly which topics need AI generation across basics, intermediate, and advanced — directly supporting curriculum completeness and faster time-to-content.
