Investor brief · 1-page summary

How EduBlast runs its Gyan++ bot
reliably without breaking the database

Plain-English explanation of why scheduled jobs can stress our database, why Vercel’s free plan is a blocker, and the cheapest reliable way to keep our AI community active every 10 minutes.

1 The problem in one picture

A scheduled job spikes the database for the same reason a busy table spikes a restaurant — too few “waiters” (database connections) for the number of people trying to eat at once.

What a database “connection” actually is

Every user, every page load, every scheduled job needs one connection to read or write. There are only ~15 free connections in our Supabase pool at any moment.

  • A real student opens a topic holds 1 connection for ~0.3s
  • A real student posts a doubt holds 1 connection for ~0.5s
  • Gyan++ bot runs a cycle holds 1–3 connections for 5–15s

Why this matters: a Gyan++ bot cycle is a big job — it calls an AI to write a doubt, calls another AI to answer it, and writes to several tables. While that happens, our free pool of 15 connections is squeezed. If it runs during peak hours, real students feel the slowdown.

2 Why Vercel’s free plan is the real blocker

Vercel is the platform that runs our backend. On its free (“Hobby”) plan, scheduled jobs can run only once per day. We need one every 10 minutes.

Hobby (current) Once per day 100 $0 Blocker
Pro Once per minute 100 $20/seat + usage Unlocks 10-min

What happens if we try to put */10 * * * * on Hobby: the deploy fails with “Hobby accounts are limited to daily cron jobs.” So today, we have no scheduled jobs at all.

3 Where the work actually runs

A scheduled job is just an HTTP request. The actual AI + database work happens on a Vercel Function. Supabase only stores the data. That separation is the whole game.

5 Our recommendation

Use Vercel Pro as the primary scheduler and keep the admin Run one cycle button as the manual fallback. Supabase spikes won’t disappear — but with Pro, each cycle is fast, predictable, and capped (e.g. 60s max), so a slow cycle can’t hold a connection forever.

Pick: Vercel Pro + admin fallback ~$20/month buys a real timer + cuts spike blast radius.

6 What we actually have to do

No code rewrite. The route already exists. We just turn on the timer and set one secret.

  1. Upgrade the Vercel team to Pro ($20/seat/month). Takes 2 minutes in Vercel billing.
  2. In Vercel project Settings → Environment Variables, add CRON_SECRET with a long random value (e.g. kbT8g2xL9vH7pQ4sR1wE5yN3mZ0aD6cF). Tick Production.
  3. Replace the empty vercel.json with: 
    {
  "crons": [
    { "path": "/api/cron/gyan-bot-post", "schedule": "*/10 * * * *" },
    { "path": "/api/cron/refund-doubt-bounties", "schedule": "0 0 * * *" }
  ]
}
  4. Commit and push to the production branch. Vercel auto-deploys and registers the cron.
  5. Test once with curl from a laptop: 
    curl.exe -i -H "Authorization: Bearer YOUR_SECRET" https://www.edublast.in/api/cron/gyan-bot-post
    Expect 200 OK with a JSON body.
  6. Watch Vercel → Logs and Supabase → doubts table for 24 hours. The last_post_at in the admin panel updates every cycle.
  7. To stop everything instantly: open Admin → Gyan Bot and click Stop. The cron keeps firing but returns skipped: bot inactive. No AI calls, no DB writes, no cost.

7 What this actually costs — we don’t know yet

Honest answer: we do not have a month of real cron traffic to measure from. Every number that would go here is a guess.

The only way to know the real number is to turn the cron on, let it run one full day, and read the bills:

  • Vercel → Usage tab on the dashboard shows compute-hours consumed by the cron function.
  • Sarvam → billing dashboard shows token usage for the AI calls the bot makes.
  • Modal → GPU-seconds for the RAG retrievals (today this is on the free tier).
  • Supabase → Reports → Database shows connection-hours and IO during the same window.

What we can predict before turning it on

  • Compute: short-lived (single-digit seconds), low memory.
  • AI: one doubt-generation call + one answer call per cycle.
  • DB: a handful of reads + a couple of writes per cycle.
  • RAG: one retrieve per cycle (free on our current Modal plan).

8 Safety nets already in the code

Even if multiple clicks or pings fire, the server is the gate. Nothing can go rogue.

🔐 Static secret (cron route)

The external route requires Authorization: Bearer <CRON_SECRET>. In production on Vercel, the route refuses with 503 if the secret is missing.

👤 Admin session (admin route)

The Run One Cycle button uses the admin’s Supabase session. Random users get 401, non-admins get 403.

⏱️ 10-min interval gate

The bot refuses to run a real cycle if the last post was less than interval_minutes ago. Returns skipped: interval not elapsed.

🛑 Master switch

Setting active = false in gyan_bot_config stops all AI work, even when the cron keeps firing.

⏳ Function timeout

Set maxDuration = 60s on the route so a stuck cycle dies fast. Releases the Supabase connection.

📍 Region pinning

Pick the Vercel function region closest to your Supabase project. Lower latency = shorter connection hold = smaller spike.

9 What we still need to watch

  • Supabase connection spike does not go away. Pro makes it shorter and more predictable, not gone. To shrink it: enable Supavisor transaction pooler in Supabase dashboard.
  • Spike days will cost more than the average. Many students hitting the bot at the same time means retries, slow AI, longer connection holds. First month is the worst month because we will be learning from it — use the master switch + interval knob to keep it predictable.
  • Single-region risk. If Vercel has an outage, the bot just skips that slot. The interval gate handles it; no data loss.
